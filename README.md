# AutoCure — Self-Healing RPA Automation Solution

AutoCure is an AI-powered, self-healing Robotic Process Automation (RPA) framework that automates candidate onboarding on a web portal. When the target application's UI changes and breaks the bot's element locators, the **AutoCure Agent** uses Google Gemini to analyze the DOM changes, regenerate resilient XPaths, and resume automation — all without human intervention.

---

## 📌 Problem Statement

Traditional RPA bots are brittle: a small change in the web page structure (renamed IDs, moved elements, redesigned layouts) breaks hard-coded locators and halts the entire automation pipeline. AutoCure solves this by:

1. **Detecting** locator failures at runtime (Selenium `TimeoutException`).
2. **Diagnosing** the change by comparing the *baseline HTML* with the *live page source*.
3. **Healing** the XML locator repository using an LLM (Gemini 2.5 Flash) that generates new, relative, resilient XPaths.
4. **Retrying** the failed record automatically with the freshly healed locators.


## 📂 Project Structure

| File | Description |
|------|-------------|
| `app.py` | Flask backend serving the onboarding portal (`/`) and the submission API (`/api/submit`). Appends each submitted candidate to `Infosys_Onboarding_Records.xlsx`. |
| `index.html` | Frontend onboarding form (Candidate ID, Full Name, Phone, PAN, Aadhaar, DOB, Address) with async JS submission and a success banner. |
| `rpa_executor.py` | The RPA engine. Reads records from `input_data.xlsx`, drives Chrome via Selenium using XPaths from `locators_v1.xml`, submits each record, and verifies the success message. On failure, it captures the live DOM and invokes AutoCure, then retries. |
| `autocure.py` | The `AutoCureAgent` class. Sends the baseline HTML, live (broken) HTML, and current XML to Gemini via the modern `google-genai` SDK, validates the returned XML, backs up the old file, and writes the healed locators. |
| `locators_v1.xml` | Central locator repository. Each `<Element>` has a stable `name` (mapped to Excel columns) and an `xpath`. AutoCure only rewrites XPaths — names are preserved to keep the Excel mapping intact. |
| `input_data.xlsx` | Input dataset the bot consumes. Column headers must match locator `name` attributes: `CandidateID`, `FullName`, `PhoneNumber`, `PANNumber`, `AadhaarNumber`, `DOB`, `Address`. |
| `Infosys_Onboarding_Records.xlsx` | Output file generated by the Flask backend containing all successfully submitted onboarding records. |
| `_env` (rename to `.env`) | Environment file holding `GOOGLE_API_KEY` for the Gemini API. |
| `index_v1.html` (baseline) | A saved snapshot of the original portal HTML used as the "known good" reference during healing. Create it by copying `index.html` before making UI changes. |

## ⚙️ Prerequisites

- Python 3.10+
- Google Chrome + a matching ChromeDriver (Selenium Manager in Selenium 4.6+ handles this automatically)
- A Google AI Studio API key (for Gemini)

### Python Dependencies

```bash
pip install flask pandas openpyxl selenium python-dotenv google-genai
```

## 🚀 Setup & Execution

### 1. Configure the API Key
Rename `_env` to `.env` and set your key:

```env
GOOGLE_API_KEY=your_actual_gemini_api_key
```

### 2. Create the Baseline Snapshot
```bash
cp index.html index_v1.html
```
This is the reference DOM that AutoCure compares against when healing.

### 3. Start the Onboarding Portal
```bash
python app.py
```
The portal runs at `http://127.0.0.1:5000/`. Manual submissions and bot submissions are both saved to `Infosys_Onboarding_Records.xlsx`.

### 4. Run the RPA Bot
In a second terminal:
```bash
python rpa_executor.py
```
The bot will:
1. Load locators from `locators_v1.xml`.
2. Read all rows from `input_data.xlsx`.
3. Open Chrome, fill the form for each candidate, click **Submit Details**, and wait for the success banner.

## 🤖 The Self-Healing (AutoCure) Workflow

1. **Failure detection** — `process_record()` raises a `TimeoutException` when any XPath cannot locate its element within 5 seconds.
2. **Live DOM capture** — the executor grabs `driver.page_source` (the actual broken page).
3. **AI analysis** — `AutoCureAgent.heal_locators()` prompts Gemini 2.5 Flash with:
   - the baseline HTML (`index_v1.html`),
   - the live broken HTML,
   - the current `locators_v1.xml`.
4. **Constraint enforcement** — the prompt requires strictly valid XML, relative XPaths, and **unchanged `name` attributes** (protecting the Excel column mapping).
5. **Validation & versioning** — the response is parsed with `xml.etree.ElementTree` to guarantee well-formed XML; the old file is backed up as `locators_v1_old_YYYYMMDD_HHMMSS.xml` before being overwritten.
6. **Hot reload & retry** — the executor reloads the healed locators into memory and retries the failed record (max 1 retry per record by default).

### Demonstrating the Self-Healing
1. Start the bot with the working portal and let a few records process.
2. Modify `index.html` (e.g., change label text, restructure divs, rename the submit button) and restart Flask.
3. On the next record, the bot fails, triggers AutoCure, regenerates the XPaths, and continues — check the logs for:
   ```
   [WARNING] Execution failed on record N due to DOM change. Triggering AutoCure...
   [INFO] AutoCure Successful: Re-generated locators_v1.xml with updated XPaths.
   [INFO] AutoCure complete. Retrying record...
   ```

## 🔑 Key Design Decisions

- **XML as a single source of truth** — no XPaths are hardcoded in Python; the bot is fully configuration-driven, so healing the XML heals the bot.
- **Name-stable locators** — locator `name` attributes double as Excel column keys, which is why AutoCure is forbidden from renaming them.
- **Low-temperature generation** (`temperature=0.1`) — makes the AI output deterministic and format-compliant.
- **Defensive parsing** — markdown fences are stripped and the XML is syntax-validated before it ever touches disk; invalid output aborts the heal gracefully.
- **Version control for locators** — every heal creates a timestamped backup, giving a full audit trail of locator evolution.
- **Localhost-only Flask** — the server binds to `127.0.0.1` with `debug=False`, keeping the demo portal off the network.

---

## ⚠️ Notes & Limitations

- Sensitive PII (PAN, Aadhaar) is stored in plain Excel files — this project is a **demo/PoC**; add encryption and access controls before any real-world use.
- The `input_data.xlsx` column headers must exactly match the locator `name` attributes, or fields will be silently skipped.
- Healing quality depends on the baseline snapshot being reasonably current; refresh `index_v1.html` after intentional UI releases.
- Max retries is set to `1` per record in `rpa_executor.py`; increase `max_retries` for more aggressive recovery.
